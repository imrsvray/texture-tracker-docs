Detailed Description

Estimates the initial pose of a camera, given a PoseEstimator object that has a set of 3D lines stored, and observations of points that may lie on lines as viewed by the camera.

The class is intended for use as follows:

  • Instantiate one instance of the class when the main program first starts, giving it a PoseEstPanTiltHead object that has already been set up with data on the lines in the world and a camera whose internal parameters (pixel size, centre point, etc) have all been set.
  • Give it information on the camera positions and ranges of movement that are expected. This can be done in one of three ways:

1) addPossibleCameraLocation() adds a location taking data from the given variables

2) addPossibleCameraLocations() adds data for multiple locations from the given text file

3) loadProcomputedSearches() loads in a pre-calculated list of all lines to search for in the full range of poses for multiple cameras

  • Call other "set" members of the class as required to set values for the various parameters used in the processing. Most default to sensible values.
  • To make a new initial estimatate of the the camera pose, call fullEstimate(), giving it pointers to the image to analyse.

Other methods are provided to allow the individual steps in fullEstimate() to be carried out separately: these methods are estimatePose()/estimatePosePrecomputed(), and refinePoses(). These are mainly intended for debugging/testing, though. Note also that the addLinesFromImageUsingEstimatedPose() method (used by refinePoses() ) may be useful for image-to-image tracking once initialisation has been carried out.

Notes about representation of camera pose:

The position of the camera is represented in metres, in a right-handed reference frame which is usually chosen to have x parallel to the longest direction of the pitch, and y vertical.

The rotation of the camera is specified in terms of pan and tilt angles in degrees, with roll assumed to be zero. When these angles are zero, the camera looks down the -ve z axis, with y vertical. Positive angles correspond to panning anticlockwise and tilting up.

The world reference frame (e.g. the reference frame of the football pitch) can be rotated, and this rotation can be different for each camera. This provides a way of representing the angle between the world vertical axis and the axis that the camera pans about (which is why the rotation can be different for each camera). The angles are represented as OpenGL-style rotation angles and are specified in degrees. Normally it is only necessary to have the rotation about the z axis as non-zero, as this is sufficient to correspond to any apparent roll in a camera whose dominant direction of view is parallel to the z axis.

Note that this initialisation class performs a search over the range of pan, tilt and roll specified, using the given values of camera position and pitch orientation. It can refine the camera position and pitch orientation during the latter stages of its processing if the appropriate adjustments are enabled in the PoseEstPanTiltHead class, but this is probably best handled externally, e.g. by using the PoseEstPanTiltHead class to compute values optimised over several images.

Search the given RGB image for lines close to the position expected based on the currently-set camera pose.

The search distances, thresholds, weighting parameters, etc. are those previously set with methods such as setRGBWeights(). The lines found are put into the current image in the PoseEstPanTiltHead object supplied in the constructor.

Parameters:

nLinesToUseThe number of lines to use (-1=use all stored lines). Useful if some lines have been added for manual calibration and we don't want to search the image for these - but make sure the lines we want to search for automatically are stored first, ie. with index starting from 0

Note that when this method is called during the initialisation process (by refinePoses(), which is called by fullEstimate()), the velocity estimate is set to zero (as no reasonable estimate is available), so the vaious motionBlurFactor parameters only have an effect when addLinesFromImageUsingEstimatedPose() is called directly, e.g. for use in line tracking rather than initialisation.

Adding all points is important if you are taking account of lens distortion, as a fitted straight line will no longer be a fair representation of the appearance of the line, but will make the pose computation take slightly longer.

int bbcvp::InitialPoseEstLines::addPossibleCameraLocations ( const char *  fileName)

Add some ranges of possible angles, field-of-view and positions to be searched from a text file.

Format of text file is multiple lines like the following:

pos lf lf lf pan lf lf lf tilt lf lf lf fov lf lf lf

where numbers after "pos" are x, y, z positions in metres, and numbers after pan,tilt,fov are min angle, max angle and step size (in pixels) to move the angle through between tests.

Lines are ignored after maxLocation locations have been added. Lines starting with "#" are also ignored.

Returns:

Set thresholds for line detection.

Used by the line finder both during initial pose estimation, and when refining the pose. Initial defaults (which work well for white lines on green grass) are set in the constructor.

Parameters:
pitchThreshMinimum value of output from ChromaKeyer that must be seen either side of a pixel for it to be considered as a possible line of interest.

lineThreshMotionBlurFactorThe effective camera integration time to assume when computing how to reduce lineThreshFullWeight in the presence of image motion. The value of lineThreshFullWeight used in addLinesFromImageUsingEstimatedPose is computed as

lineThreshZeroWeight + (lineThreshFullWeight-lineThreshZeroWeight) * width_pixels / blurWidth

where width_pixels is the expected width across the line (derived from the projection of pitchLineWidth into the image), and blurWidth is the motion speed at right angles to the line (derived from the the image velocity velX,velY at right-angles to the direction of the line being searched for, multiplied by lineThreshMotionBlurFactor). The idea is that motion blur will reduce the expected 'strength' of a line, so the value needed for a point to be fully-weighed should be reduced as the motion speed increases. If lineThreshMotionBlurFactor is 1.0, the computation will reflect the expected behaviour for a camera with no shuttering, and the units of velX,velY being pixels per field period. If lineThreshMotionBlurFactor=0, a camera with a very short shutter time (no motion blur) is assumed. This is the default.
